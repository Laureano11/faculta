Rta: Falso. La experiencia del analista y su subjetividad pueden ser una desventaja.



7) Un profesional de sistemas siempre debe resolver problemas bajo el enfoque sistémico.

9) Uno de los objetivos de la metodología de sistemas es reducir el costo de mantenimiento.

Rta: Verdadero. Es una de las razones, pero no la única.


10) Todos los requisitos son determinados por el usuario.

Rta: Verdadero. Tanto los funcionales como los no funcionales.


11) La calidad de un producto software está asociada al uso de tecnología informática de última generación.

Rta: Falso. El método o técnica que se utilice en la construcción del software proporciona el criterio de calidad, además de las herramientas para usar dicho método. También la aplicación de estrategias de prueba y mecanismos de medición.


12) No existen sistemas de información impermeables.

Rta: Verdadero. El sistema de información es un sistema abierto, o sea, es permeable. Tiene interacción constante con el medio.


13) Todo cambio que se realice durante la etapa de codificación implica una modificación de los modelos conceptuales. 

Rta: Verdadero, para tener actualizado el modelo.


14) Las etapas de un ciclo de vida de un sistema pueden alterarse totalmente cuando se trate del desarrollo de un sistema orientado al apoyo de decisiones.

Rta: Falso, las etapas no varían.


16) El estudio de factibilidad en el desarrollo de un sistema es una etapa en la que se puede analizar en forma clara y precisa el entorno de la inversión a realizar por la empresa.

Rta: Falso. Se evalúan y proponen distintas alternativas de solución a los requerimientos y se sugiere el mejor curso de acción, utilizando diversos criterios, como el económico, político, legal, etc.


17) La portabilidad de un software se caracteriza porque permite adaptar el modelo lógico de un sistema a cualquier plataforma de hardware y entorno de sistema operativo.

Rta: Verdadero. Por definición de portabilidad.


18) Las tablas de decisión permiten analizar los casos que no tienen solución en un DFD. Esto es así porque la TD es una técnica que permite un mayor grado de detalle que el DFD, en el modelo lógico de desarrollo.

Rta: Falso. Ambas herramientas sirven para el modelado de procesos, por eso tienen el mismo grado de detalle.



21) Un sistema de información que interactúe con al menos 5 usuarios debe definirse como un sistema software.

Rta: Falso. La necesidad de tecnología informática no está ligada con la cantidad de personas que usen el sistema. Si un sistema funciona perfectamente de forma manual, no habrá necesidad de incluir software.



23) El cursograma es complementario a la tabla de decisión ya que los dos reflejan la misma información.

Rta: Falso. Ambas son herramientas utilizadas para documentar información en la etapa de relevamiento, pero el cursograma muestra cómo circula la documentación y la tabla de decisión muestra las distintas decisiones que pueden tomarse respecto a un problema.


24) Los requerimientos funcionales se determinan de forma óptima si se utiliza la metodología orientada a objetos porque se pueden definir distintos diagramas.

Rta: Verdadero. El modelado del análisis orientado a objetos permite una mejor captura y validación de requisitos.


25) El usuario responsable de suministrarle la información para realizar el sistema, dice que no tiene tiempo para realizar las entrevistas durante el Relevamiento. Considera Ud. que mediante la observación puede obtener toda la información necesaria para realizar el sistema encomendado.

Rta: Verdadero. Ambas son técnicas de obtención de información. Consiste en observar al que toma decisiones y su lugar de trabajo. Se busca obtener datos estadísticos de cuál es la información que maneja, cómo la obtiene y comparte.


26) El criterio de productividad en el desarrollo de software debe definirse durante la etapa de implementación. Esto permite finalizar dicha etapa lo antes posible, y asi dejar funcionando el sistema en menos tiempo.

Rta: Falsa. En la etapa de relevamiento, es un requisito no funcional.


27) Las Tablas de Decisión permiten encerar en forma ordenada y estructurada todo tipo de decisión, dentro de la organización.

Rta: Falso. Sólo es útil para decisiones programadas, que se toman a nivel operativo. Además, no dice nada acerca del orden de toma de decisión. 


28) Al momento de realizar el desarrollo del código (etapa de programación), los programadores le solicitan al analista realizar cambios en las especificaciones del sistema, basados en el objetivo de acelerar el desarrollo del mismo. Rta: Falso. Nada indica que será así, ya que quizás las etapas previas a la programación no se adecúen a las nuevas especificaciones. Puede implicar algún tipo de relevamiento y análisis nuevo.


29) La gestion de calidad de un proyecto de software comienza cuando se inicia el diseño del mismo.

Rta: Falso. Se realiza en toda etapa del proceso de desarrollo.


30) Cuánto más especializado es un sistema, menor será su adaptabilidad a las diferentes circunstancias.

Rta: Falso. La adaptabilidad de un sistema es la propiedad de aprender y modificar un proceso, estado o característica de acuerdo a las modificaciones que sufre el contexto. Se relaciona con el concepto de homeostasis, que es la capacidad de supervivencia del sistema a través de su adaptación al contexto.


31) El criterio de modularidad en el desarrollo de un software se relaciona con el enfoque sistémico de división del sistema en subsistemas facilitando su comprensión y evaluación.

Rta: Falso. Se relaciona con el enfoque orientado a objetos. Es la capacidad del sistema de ser visto como la unión de parte que interactúan entre sí en busca de un objetivo en común.


32) El árbol de decisión es preferible a la tabla de decisión para analizar casos muy complejos.

Rta: Verdadero. Se utiliza para ramificaciones complejas en los procesos estructurados de decisión. El mismo no es binomial


33) Cuando se analizan los costos de mantenimiento de un sistema, algunos de los factores que se consíderan son el costo de adquisición del hardware y cuál será el retorno de inversión.

35) La gestión de calidad de un proyecto de software comienza cuando se inicia el desarrollo del mismo.

Rta: Verdadero. Desde el método o técnica para el desarrollo del software se encuentra apoyado en fuertes principios de calidad, los cuales son subjetivos.


36) En un diagrama de Gantt, si una tarea comienza inmediatamente después que otra haya finalizado no se puede asegurar que ésta última dependa de la primera.

Rta: Falso. El diagrama de Gantt no permite ver la interdependencia entre tareas.


37) La solución tecnológica de un proyecto de software se define durante la etapa de análisis funcional.

Rta: Falso. Se define en la etapa de estudio de factibilidad, cuando se selecciona la alternativa de solución.
